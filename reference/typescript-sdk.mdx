Vaulkyrie has a TypeScript SDK under `src/sdk/`, but it is currently internal to the browser wallet. It is not yet a standalone npm package.

## Current exports

`src/sdk/index.ts` exports:

- `constants`
- `types`
- `pda`
- `instructions`
- `accounts`
- `errors`
- `VaulkyrieClient`

## What it can do today

| Capability | Source |
| --- | --- |
| PDA derivation | `src/sdk/pda.ts` |
| Instruction builders | `src/sdk/instructions.ts` |
| Account decoders | `src/sdk/accounts.ts` |
| Error parsing | `src/sdk/errors.ts` |
| Account fetch helpers | `src/sdk/client.ts` |
| Spend binding hashes | `src/sdk/spendBindings.ts` |

## Builder coverage

The internal TS SDK currently includes builders for:

- `createInitVaultInstruction`
- `createInitAuthorityInstruction`
- `createInitPqcWalletInstruction`
- `createAdvancePqcWalletInstruction`
- `createSetVaultStatusInstruction`
- `createInitSpendOrchestrationInstruction`
- `createCommitSpendOrchestrationInstruction`
- `createCompleteSpendOrchestrationInstruction`
- `createInitRecoveryInstruction`
- `createCompleteRecoveryInstruction`

It does not yet expose every Rust SDK builder. Missing public TS builders include authority proof chunking, staged authority rotation, winter authority advance, quantum vault split/close, spend failure, and authority migration.

## Example: fetch a vault registry

```ts
import { Connection, PublicKey } from "@solana/web3.js";
import { VaulkyrieClient } from "@/sdk";

const connection = new Connection("https://api.devnet.solana.com", "confirmed");
const client = new VaulkyrieClient(connection);

const vault = await client.getVaultRegistry(new PublicKey(walletPublicKey));
```

## Example: initialize a PQC wallet instruction

```ts
import { PublicKey } from "@solana/web3.js";
import { createInitPqcWalletInstruction } from "@/sdk";

const ix = createInitPqcWalletInstruction(
  new PublicKey(payer),
  new PublicKey(pqcWalletPda),
  {
    walletId,
    currentRoot,
    bump,
  },
);
```

## Example: build spend orchestration bindings

```ts
import {
  buildSpendActionHash,
  buildSpendOrchestrationBindings,
  generateSpendSessionNonce,
} from "@/sdk/spendBindings";

const sessionNonce = generateSpendSessionNonce();
const actionHash = await buildSpendActionHash({
  vaultId,
  recipient,
  amountAtomic: "1000000",
  tokenSymbol: "SOL",
  tokenMint: null,
  sessionNonce,
});

const bindings = await buildSpendOrchestrationBindings({
  actionHash,
  messageBytes,
  signerIds: [1, 2],
  threshold: 2,
  participantCount: 3,
  expirySlot: 123456n,
});
```

## Should Vaulkyrie create a standalone TypeScript SDK?

Yes, if external developers are expected to integrate Vaulkyrie into their own codebases.

The current `src/sdk/` is useful, but it is coupled to the browser wallet project structure and not packaged. A standalone SDK should be created before telling third-party developers to install and import it.

Recommended next implementation:

1. Create a dedicated package such as `packages/vaulkyrie-sdk-ts`.
2. Move code that has no extension dependency out of `src/sdk/`.
3. Add a package manifest with ESM output and type declarations.
4. Add fixtures that compare TypeScript instruction bytes against `crates/vaulkyrie-sdk`.
5. Close builder parity gaps with the Rust SDK.
6. Publish or document local package linking only after tests pass.

## Current readiness

| Question | Answer |
| --- | --- |
| Is there TS SDK code? | Yes, in `src/sdk/`. |
| Is it published as npm package? | No. |
| Is it complete against Rust SDK? | No. It covers the browser's current needs, not every builder. |
| Can docs include snippets? | Yes, but they should be labeled as internal imports until extraction. |
